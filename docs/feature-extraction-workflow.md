# 患者自然语言到分诊特征的 Workflow

## 1. 这版 Workflow 的重点

当前版本重点不再只是“有没有分诊结果”，而是：

- 患者说了一句通俗的话
- 系统如何从里面提取关键词
- 如何把口语转换成结构化 feature
- 如何把这些 feature 交给规则引擎
- 最后如何生成分诊等级和下一轮追问

## 2. 当前采用的特征提取流水线

### Step 1. 保留患者原话

患者每次回复都会先原样写入 `conversation_history`。

这样做的意义是：

- 后续规则引擎仍然可以回看原话
- LLM 也能结合上下文继续理解
- 前端可以展示“原话 -> 结构化结果”的对应关系

### Step 2. 正则提取

系统先找最容易稳定提取的数值或显式字段，例如：

- 体温 `38.5度`
- 疼痛评分 `7/10`
- 持续时间 `2天`

这一层最适合做：

- 数字
- 时间长度
- 明确量化信息

### Step 3. 关键词提取

系统再找明显症状词，例如：

- `腹痛`
- `胸痛`
- `呼吸困难`
- `呕吐`
- `头晕`

这一层适合做：

- 标准医学症状词
- 外伤词
- 严重程度词

### Step 4. 通俗表达映射

这是这次 workflow 特别强调的部分。

患者往往不会直接说标准医学术语，而是说：

- `喘不上气`
- `胸口发闷`
- `肚子绞着疼`
- `一直想吐`
- `犯迷糊`
- `口干得厉害`

系统会把这些口语映射成 feature，例如：

- `喘不上气` -> `呼吸困难`
- `肚子绞着疼` -> `腹痛` + `重度`
- `犯迷糊` -> `嗜睡`
- `口干得厉害` -> `极度口渴`

这一层特别适合中文医疗场景，因为患者真实表达通常非常口语化。

### Step 5. LLM 补充结构化抽取

如果百炼可用，系统还会调用 LLM 对本轮消息做补充抽取。

LLM 负责：

- 补充漏掉的症状
- 识别伴随症状
- 提示疑似风险信号
- 提供更自然的下一轮追问

### Step 6. 合并提取结果

系统把下面几层结果合并：

- 正则提取
- 关键词提取
- 通俗表达映射
- LLM 抽取

最后生成统一的结构化更新，例如：

```json
{
  "体温": 38.5,
  "起病时间": "昨天",
  "识别症状": ["腹痛", "恶心"],
  "伴随症状": ["恶心"],
  "严重程度": "重度"
}
```

### Step 7. 规则引擎复核

合并后的 feature 会进入规则引擎。

规则引擎负责：

- 红旗征象判断
- 红黄绿分级
- 隐性危重识别
- 易混淆科室边界防错
- 强制体温校验

### Step 8. 生成下一轮追问

在不是红区的情况下：

- 优先由 LLM 生成下一轮追问
- 如果 LLM 不可用，则回退到模板追问

## 3. 当前代码中由谁负责哪一层

- `app/services/feature_extractor.py`
  - 负责正则提取、关键词提取、通俗表达映射、LLM 抽取合并、提取轨迹

- `app/services/triage_service.py`
  - 负责把提取结果写回 Session Context，并继续推动会话

- `app/rule_engine.py`
  - 负责规则判断和分诊输出

- `app/services/llm_service.py`
  - 负责百炼结构化抽取和下一轮追问生成

## 4. 当前 UI 为什么有用

新增的 `/demo` 页面不只是为了“更好看”，它主要是为了把整个 workflow 可视化：

- 左边：患者首轮输入
- 中间：多轮对话
- 右边：结构化更新、提取轨迹、分诊结果

所以这个 UI 的价值是：

- 能看见原话
- 能看见 feature
- 能看见规则命中
- 能看见分诊输出

## 5. 下一步还能继续加强什么

最值得继续增强的方向：

1. 扩大中文口语表达映射词库
2. 增加部位、性质、诱因等更细特征
3. 用更多病例做 feature 抽取测试
4. 把提取轨迹和最终交接单一起保存到数据库
