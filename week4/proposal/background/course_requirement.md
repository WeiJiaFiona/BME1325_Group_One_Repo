# 《模拟医院》课程说明 (BME1325)

## 课程基本信息

| 项目 | 内容 |
|------|------|
| 课程名称 | 模拟医院 (Sims Hospital) |
| 课程代码 | BME1325 |
| 教学形式 | 项目制学习 (Project-Based Learning) |
| 考核方式 | 过程评估 + 最终答辩 |
| 学生组织 | 3 人小组 |

---

## 课程目标

通过构建**多智能体医院模拟系统**，让学生：
1. 理解医疗流程与复杂场景中的决策逻辑  
2. 掌握大模型智能体的设计与实现方法  
3. 培养系统思维和工程实践能力  
4. 思考 AI 医疗应用中的伦理与安全问题  

---

## 四阶段学习路线

### 阶段一：场景理解与需求识别

**目标**：理解医院运作方式，从实际场景中识别 AI 智能体能解决的问题。

**学习内容**
- 医院典型流程：门诊、急诊、住院、手术室、医技科室
- 场景分析方法：流程图、角色、信息流、瓶颈识别
- 需求转化：将"场景问题"转化为"智能体任务"
- 大模型智能体基础：Prompt、Memory、Tool、Policy、Environment

**阶段交付**
- 个人场景观察报告
- 小组智能体任务定义（如：分诊 Agent、问诊 Agent、护理 Agent 等）

---

### 阶段二：大模型智能体建模

**目标**：掌握大模型智能体的核心设计模式，构建可独立运行的医疗角色智能体。

**学习内容**
- Agent 架构：ReAct 范式、Tool Use、Workflow 设计
- Prompt 工程：角色设定、任务约束、交互风格
- Memory 机制：患者状态、对话历史、长期信息
- Tool 调用：RAG 知识库检索、数据库查询、流程调度
- 医疗角色建模：医生、护士、患者、行政人员等角色特性

**阶段交付**
- 2–3 个可独立运行的智能体（含单体测试）
- 设计文档：Prompt、Memory 结构、Tool 列表

---

### 阶段三：多智能体系统集成与场景模拟

**目标**：将多个智能体整合到医院环境中，形成系统级的互动与协作。

**学习内容**
- 医院环境建模：状态机、事件驱动、资源管理
- Agent–Environment 交互：状态广播、任务分配、反馈机制
- 多智能体协作：消息传递、角色对话、协调决策
- 场景设计：完整的门诊/急诊流程、突发事件处理
- 可观测性：日志、监控、运行轨迹可视化

**阶段交付**
- 完整的医院子系统（门诊 OR 急诊）及多智能体协作演示
- 中期报告：系统架构、关键设计、初步测试结果

---

### 阶段四：系统优化、安全与最终交付

**目标**：完善系统性能与可靠性，深入思考伦理与安全，准备最终展示。

**学习内容**
- 可靠性与安全：大模型幻觉、错误决策、出错恢复策略
- 医疗伦理：患者隐私、知情同意、算法偏见控制
- 性能优化：响应延迟、吞吐量、资源效率
- 系统完善：边界条件、异常处理、用户体验
- 技术传播：项目报告、系统演示、答辩准备

**阶段交付**
- 完整的"模拟医院"系统（多个子系统协作）
- 最终技术报告（需求 → 设计 → 实现 → 测试 → 伦理）
- 小组答辩与系统演示视频

## 分诊智能体（Triage Agent）—— 技术示例

我们以**分诊护士智能体**为例，展示从 Prompt 到运行的完整流程。

### 1. 智能体的角色定义（System Prompt）

```
你是一名医院急诊科的分诊护士智能体（Triage Agent）。

你的职责：
1. 与患者进行友好、专业的对话，收集症状、病史、严重程度
2. 调用医学知识库工具，查询症状对应的危险信号
3. 根据医院分诊标准（红/黄/绿）给出分诊等级
4. 将分诊结果写入医院系统
5. 所有回答必须简洁、易懂、具同理心

禁止事项：
- 进行诊断（如"你得了肺炎"）
- 给出治疗方案
- 作出医学承诺
```

### 2. 智能体的记忆结构（Memory）

```json
{
  "patient_profile": {
    "age": null,
    "gender": null,
    "chief_complaint": "主诉症状",
    "symptoms": ["症状1", "症状2"],
    "duration": "症状持续时间",
    "severity": "1-10 评分",
    "vital_signs": {}
  },
  "conversation_history": [
    {"role": "nurse", "content": "..."},
    {"role": "patient", "content": "..."}
  ]
}
```

**Memory 的作用**
- 保持对话一致性（Agent 不会重复提问）
- 指导下一步提问（缺少哪些信息）
- 支持分诊决策（基于完整的患者信息）

### 3. 智能体可调用的工具（Tools）

#### 工具 A：医学知识库检索（RAG）
```python
def medical_knowledge_query(query: str) -> dict:
    """
    输入症状关键词，返回：
    - 危险信号指标
    - 相关疾病列表
    - 临床指南摘要
    """
    # 返回示例
    return {
        "dangerous_signs": ["意识改变", "呼吸困难"],
        "related_diseases": ["急性冠脉综合征", "肺炎"],
        "guidelines": "..."
    }
```

#### 工具 B：写入医院记录
```python
def write_triage_record(patient_id: str, triage_level: str, summary: str) -> bool:
    """
    将分诊结果持久化到医院系统
    """
```

#### 工具 C：流程调度
```python
def dispatch_patient(level: str, patient_id: str) -> str:
    """
    根据分诊等级安排后续流程：
    - 红色：立即急救
    - 黄色：优先挂号
    - 绿色：普通门诊
    """
```

### 4. 智能体的决策逻辑（Policy）

Agent 根据 Memory 中的患者信息 + RAG 返回的危险信号来判断分诊等级：

| 分诊等级 | 描述 | 危险信号示例 |
|----------|------|-----------|
| **红色** | 危急 | 呼吸困难、意识障碍、胸痛伴出汗、持续呕血 |
| **黄色** | 紧急 | 高烧不退、剧烈疼痛、持续呕吐、脱水迹象 |
| **绿色** | 普通 | 轻微咳嗽、轻度腹痛、轻微外伤 |

### 5. 对话示例

**患者输入**
```
"我胸口痛了两个小时，有点喘不过气。"
```

**Agent 内部流程**
1. 记录症状到 Memory
2. 调用 RAG：查询"胸痛 + 呼吸困难"
3. RAG 返回：存在"急性冠脉综合征"危险信号
4. Policy 决策：→ 红色分诊等级
5. 调用工具：write_triage_record() + dispatch_patient()

**Agent 输出（给患者）**
```
根据你描述的症状，你目前属于【红色分诊等级】，
需要立即接受急诊处理。我已经为你安排急救团队，
请跟随医护人员前往急救室。祝你早日康复。
```

### 6. 最小可运行示例（Python 伪代码）

```python
from llm_agent import LLM, Agent, Tool

# 初始化大模型
llm = LLM(model="qwen-medical")

# 注册工具
rag_tool = Tool(
    name="medical_knowledge_query",
    description="查询医学知识库",
    func=medical_knowledge_query
)
write_tool = Tool(
    name="write_triage_record",
    func=write_triage_record
)

# 创建分诊 Agent
triage_agent = Agent(
    llm=llm,
    system_prompt=TRIAGE_PROMPT,  # 上面定义的 System Prompt
    tools=[rag_tool, write_tool],
    memory={}  # 动态维护
)

# 运行对话
response = triage_agent.chat(
    user_message="我胸口痛了两个小时，有点喘不过气。"
)
print(response)
```

## 学生所需的技能清单

### 第一层：编程与工程基础（第 1–2 周必备）
- ✓ Python 基本语法（变量、函数、类、模块）
- ✓ 虚拟环境管理（venv / conda）
- ✓ 包管理（pip / poetry）
- ✓ 基本调试（print、日志、错误处理）
- ✓ Git 基础（clone、commit、push）

### 第二层：大模型智能体核心（第 2–5 周重点）

**Prompt 工程**
- 理解 System Prompt、User Prompt 的作用
- 编写清晰的角色定义、任务约束、输出格式
- 多轮对话中的上下文管理
- Prompt 模板化（如 Jinja2）

**Memory 机制**
- 短期记忆：对话上下文窗口
- 长期记忆：患者状态、病例历史
- 代码实现：dict / JSON 结构维护

**Tool Use（工具调用）**
- 函数签名与文档规范
- Tool 的注册与调用
- RAG 基础：向量检索 → 知识库查询
- 与外部系统的数据交互（API、数据库）

**Agent Workflow**
- ReAct 循环：Thought → Action → Observation
- 多工具链式调用
- 任务分解（Task Decomposition）
- 错误处理与重试策略

### 第三层：系统开发（第 6–8 周）

**环境建模**
- 状态机设计（状态 → 事件 → 下一状态）
- 事件队列与事件驱动架构
- 资源管理（队列、床位、医护人员）

**多智能体交互**
- 消息传递机制
- 任务分发与协调
- Agent 间的协作与冲突解决

**系统框架（可选）**
- FastAPI / Flask：构建简单 API
- Streamlit / Gradio：快速构建交互界面
- 日志与监控系统

### 第四层：专项（第 9–12 周）

**可靠性与安全**
- 处理 LLM 幻觉（Hallucination）
- 输出验证与纠正
- 故障恢复策略

**医疗伦理与法规**
- 患者隐私保护（去标识化、数据加密）
- 知情同意与透明度
- 算法偏见检测与缓解

**文档与展示**
- 需求文档撰写
- 系统架构设计图
- 技术报告与答辩

## 常见问题

**Q1: 我不是医学专业, 能完成这个项目吗？**  
A: 完全可以。医疗知识可以通过知识库 (RAG) 和 Prompt 来弥补，重点在于 AI Agent 架构设计。

**Q2: 没有大模型 API，怎么办？**  
A: 课程支持开源模型（如 Qwen、LLaMA）或申请学术优惠额度（OpenAI、阿里等）。

**Q3: 小组分工如何安排？**  
A: 建议分为：Prompt 工程师、Backend 工程师、System Architect、文档与演示等。

**Q4: 如果项目遇到卡顿，有什么办法？**  
A: 及时与导师沟通，可以降低系统复杂度、减少 Agent 数量、简化场景范围。

## 提交与评分标准

### 过程评分（50%）
- 阶段一：需求文档完整性与深度
- 阶段二：Agent 代码质量与单体测试覆盖率
- 阶段三：系统架构合理性与协作机制
- 阶段四：文档完善度与演示效果

### 最终评分（50%）
- 系统功能完整性：是否实现了预定的医疗流程
- 代码质量：可读性、可维护性、异常处理
- 创新点：有无独特的设计或算法优化
- 伦理思考：对 AI 医疗应用的深度思考
- 答辩表现：技术理解、应答能力、团队协作体现
