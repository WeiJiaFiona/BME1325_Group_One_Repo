# 模拟医院多智能体系统调研与方案思考

## 一、文档目的

本文档用于总结当前围绕模拟医院多智能体系统（Medical Multi-Agent System, MAS）的调研结果，重点包括：

- 对参考论文 **MedAgentSim** 的系统性梳理
- 对论文中关键模块的理解与澄清
- 对本项目可行实现路径的分析
- 对现有 API 模型能力与项目适配性的判断

本文档旨在为后续项目选型、系统架构设计、创新点提炼与 baseline 搭建提供依据。

---

## 二、参考论文梳理：MedAgentSim

### 2.1 论文基本信息

**论文标题**：MedAgentSim: Self-Evolving Multi-agent Simulations for Realistic Clinical Interactions

该论文提出了一个面向临床交互场景的多智能体模拟系统，其目标不是做静态医学问答，而是模拟更接近真实医疗场景的**医生—患者—检查**三方交互流程。论文强调真实临床并不是一次性给出完整病情，而是通过多轮问诊、逐步获取检查结果后再形成诊断结论。

---

### 2.2 Motivation / Observation

该论文的核心观察包括以下几点：

1. 现有很多医疗大模型评测仍以**静态问答**为主，即一开始就向模型提供完整病例信息，再要求直接作答。这与真实临床流程不一致。

2. 真实临床诊疗通常是一个**动态取证过程**：患者起初不会一次性说清楚全部信息，医生需要反复问询、逐步澄清，并在必要时主动申请检查。

3. 既有部分模拟系统虽然能够实现环境交互或角色扮演，但很多仍默认医生预先获得完整医疗信息，或缺乏开源可复现实现。

基于此，论文希望建立一个更真实的多智能体医院仿真框架，使得诊疗行为建立在**多轮交互、主动检查请求与历史经验复用**之上。

---

### 2.3 Contribution / Innovation

该论文的主要创新可以概括为以下三点：

#### （1）构建了真实感更强的临床交互模拟环境
系统不再局限于问答 benchmark，而是构建了一个可视化模拟环境，让 doctor agent 与 patient agent 在交互中逐步形成诊断过程。

#### （2）引入 doctor–patient–measurement 三方结构
系统将检查环节显式抽象为独立的 measurement agent，使得检查结果不会默认暴露，而是必须由 doctor 主动请求后才能获取。

#### （3）引入经验回放与多 agent 协作增强
doctor agent 不仅可以进行多轮问诊，还结合了经验检索、COT 推理、多医生讨论与多数投票等机制，以逐步改进系统在临床任务中的表现。

---

## 三、方法与系统架构梳理

### 3.1 整体 workflow

论文总体流程分为两个阶段：

#### （1）Conversation Phase
doctor 与 patient 进行多轮问询，并根据需要请求 measurement agent 返回检查结果，如体温、血压、ECG、X-ray 等。doctor 在不断补充证据的过程中完成诊断。

#### （2）Experience Replay Phase
系统将成功病例和错误修正后的反思经验存储起来，供未来遇到相似病例时检索调用，从而辅助当前推理。

---

### 3.2 Agent 组成

论文中的核心 agent 包括：

- **Patient Agent**：提供主诉、症状与病史，回答 doctor 提问
- **Doctor Agent**：负责问诊、推理、决定是否申请检查、给出诊断
- **Measurement Agent**：在收到请求后返回检查结果或影像结果，不主动参与诊断决策

这一结构非常适合作为模拟医院系统中的交互骨架。

---

### 3.3 Memory 设计

论文没有仅仅使用普通对话上下文，而是构建了两类长期经验缓存：

- **Medical Records Buffer**：保存成功诊断的完整病例
- **Experience Records Buffer**：保存失败后纠正得到的反思信息

新病例到来时，系统通过相似度检索召回相关历史案例，将成功经验或反思经验注入当前推理过程。

这一点对于本项目非常有借鉴意义，因为它说明医院 MAS 的“memory”不应只理解为聊天历史，而应扩展为**成功案例库 + 失败修正库**。

---

### 3.4 Tools / Planning / Skills

虽然论文没有直接使用 MCP、skills 等更近年的术语，但从系统能力来看，它已经具备典型的工具化工作流，包括：

- 请求体温、血压、ECG 等检查
- 请求 MRI / X-ray 等影像
- 检索相似历史病例
- 记录经验并进行回放

从项目实现角度，可以将其抽象为：

- **Tools**：检查申请、结果查询、病历写入、相似病例检索
- **Planning**：问诊 → 申请检查 → 获取结果 → 检索经验 → 协作汇总 → 输出诊断
- **Skills**：信息采集、证据整合、风险判断、协作仲裁

## 五、论文实验结果与系统评测方式

### 5.1 最终呈现结果

论文最终展示了三类结果：

1. **系统行为展示**：doctor / patient / measurement 在仿真环境中进行动态交互，逐步形成诊断过程。

2. **benchmark 结果**：在 NEJM、MedQA、MIMIC-IV 等任务上优于 baseline。

3. **ablation 与 bias 分析**：分析 measurement、memory、COT、ensembling 等模块的增益，并评估模型在 bias 条件下的波动情况。

---

### 5.2 如何验证系统完善性与可行性

论文给出的检验思路包括：

- 任务准确率
- 模块 ablation
- human study
- bias sensitivity 分析

对于本项目，可以进一步扩展为：

- 流程完成率
- 高危漏检率
- 多 agent 冲突率
- 平均交互轮数
- 平均工具调用次数

- 安全拦截率
- 失败模式分布

这类指标更适合模拟医院系统，而不仅仅是医学问答系统。它们也更符合老师对 workflow、状态机、资源管理、安全与伦理的要求。

## 六、围绕论文的后续问题总结

### 6.1 Measurement Agent 的真实职责是什么

需要明确的是，**measurement agent 不是负责判断“是否要拍片/做检查”的 agent**。  
判断是否需要进一步检查的主体仍然是 **doctor agent**。measurement agent 的职责是：

- 接收 doctor 的检查请求
- 返回相应检查结果
- 更新状态

换言之，它更像“检验科 / 影像科工具代理”，而不是诊断代理。

---

### 6.2 影像采集过程如何模拟

在这类系统中，通常不会真的控制医疗设备，而是采用**结果级仿真**。常见方式包括：

#### （1）预置病例 + 预置影像结果
每个病例预先存储隐藏字段，例如：

- 是否有影像
- 影像报告文本
- 图片路径

doctor 请求后，measurement agent 再暴露这些信息。

#### （2）模板生成检查结果
根据病例标签自动生成结构化影像报告，如“胸片示右下肺浸润影”。

#### （3）混合式方案
对重点 demo 病例使用真实图片，对普通病例仅返回文本报告。

对于课程项目，最推荐的是先用**文本化影像报告**完成主流程。

---

### 6.3 LLM 是否会根据影像继续输出诊断结果

会，但具体取决于输入形式。

- 若影像以**文本报告**形式返回，则 doctor agent 可直接结合报告与病史进行推理。
- 若系统支持真正的图像输入，则也可进一步发展为多模态诊断。

论文中提到在 visual language tasks 上使用了视觉编码方式，这说明其系统并不局限于纯文本输入。  
不过对本项目而言，最现实的路径仍然是先做**报告级 grounding**，而不是直接追求端到端影像诊断。

---

### 6.4 什么是 grounding 与 reasoning

在 agent 系统中，这两个概念需要严格区分：

#### Grounding
指将模型输出建立在**外部可验证证据**之上，例如：

- 患者陈述
- 检查结果
- 影像报告
- 病历信息
- 环境状态

其核心是回答“**依据是什么**”。

#### Reasoning
指模型基于这些已获得证据进行：

- 风险判断
- 诊断支持
- 决策规划
- 多 agent 协调

其核心是回答“**怎么得出这个结论**”。

在模拟医院 MAS 中，理想流程应是：

**先 grounding，再 reasoning，再行动。**

---

### 6.5 NEJM / MedQA / MIMIC-IV 是什么 benchmark

这三者并非同一种数据源：

#### NEJM
通常指从《新英格兰医学杂志》临床案例栏目整理出来的复杂病例 benchmark，更偏真实临床诊断推理。

#### MedQA
医学考试题 benchmark，通常为多项选择题，更偏知识记忆与考试型推理。

#### MIMIC-IV
真实电子病历数据库，包含 ICU 和住院相关的大规模临床数据，更偏真实病历结构与临床数据建模。

因此：

- **MedQA** 更偏“会不会做医学题”
- **NEJM** 更偏“会不会处理复杂临床病例”
- **MIMIC-IV** 更偏“能不能利用真实病历数据”

---

### 6.6 什么是 memory replay

memory replay 更常见的理解是 **experience replay**。  
其核心并不是单纯保存聊天记录，而是：

- 存储历史病例
- 记录成功路径与失败修正过程
- 在新病例到来时检索相似经验
- 将历史经验注入当前决策

因此它更接近：

**病例复盘库 + 相似案例召回机制**

而不是普通会话 memory。

在医院 MAS 中，memory replay 特别适合拆成：

- 成功案例库
- 错误修正库
- 高风险案例库

## 七、现有 API 模型是否足以完成论文相关任务

### 7.1 当前可用模型

目前可用的模型接口包括：

- GPT-5.1
- Qwen3
- DeepSeek R1

---

### 7.2 可以完成的核心任务

基于这些模型，可以完成论文中的大部分关键任务，包括：

- doctor / patient / measurement 的多轮对话
- doctor 判断是否补问、是否开检查
- 结构化输出
- tool calling
- experience replay 后的相似病例辅助推理
- reviewer / arbiter 多 agent 复核机制

其中，Qwen 和 DeepSeek 的官方文档明确支持 function/tool calling，GPT-5 系列也面向 agentic workflows。

---

### 7.3 需要降级实现的部分

当前更适合降级实现的部分包括：

- **真正基于医学图像的端到端视觉诊断**
- **完整复现论文中的图形化仿真环境**
- **严格复现论文 benchmark 数值**

更现实的做法是：

- measurement / imaging agent 返回**文本化影像报告**
- 前端展示影像缩略图或样例图
- doctor agent 主要基于文本报告做推理

---

### 7.4 更合理的模型分工建议

可以考虑采用分层协作方案：

#### GPT-5.1
适合：

- doctor agent
- reviewer / arbiter
- 复杂病例总结

#### Qwen3
适合：

- 路由 agent
- 表单采集 agent
- 工具调用型轻量 agent

#### DeepSeek R1
适合：

- triage reasoning
- reviewer / safety reasoning
- 长链推理与证据充分性检查

这样可兼顾性能、成本与系统稳定性。

## 八、对本项目的直接指导意义

综合上述调研，MedAgentSim 对本项目的主要启发包括：

### 8.1 可直接借鉴的部分

- doctor–patient–measurement 三方交互骨架
- 按需检查而非一次性暴露全部信息
- memory replay 双缓存思路
- 多 agent 协作与投票机制
- 可视化仿真展示形式

### 8.2 不应直接照搬的部分

- 把系统仅做成“临床交互模拟器”
- 忽略医院环境层的状态机、队列与资源约束
- 只关注 benchmark accuracy，而忽略流程与安全性

### 8.3 本项目仍需自行补充的关键模块

为了满足老师给出的项目要求，本项目在借鉴论文交互骨架的同时，还应增加：

- 分诊状态机
- 候诊队列与床位管理
- 检查资源调度
- 异常事件注入
- 安全监督 agent
- 日志与失败模式分析
- 隐私与伦理约束机制

---

## 九、当前阶段的结论

当前的判断是：

1. **MedAgentSim 可以作为本项目的“临床交互与多 agent 推理 baseline”**。  
2. **它不足以直接充当完整的“医院运营系统 baseline”**。  
3. **你现有的 GPT-5.1、Qwen3、DeepSeek R1 API 足以支撑该论文所涉及的大部分核心任务**。  
4. **本项目最合理的路线是：借鉴 MedAgentSim 的交互骨架，自行补足医院环境、资源管理、安全监督与系统评测机制。**

## 十、后续建议

建议接下来按以下方向推进：

### 10.1 系统架构层
明确区分三层：

- 轻量 agent 执行层
- 大模型推理层
- 安全监督层

### 10.2 模块设计层
优先定义：

- patient / doctor / measurement / safety / scheduler 几类 agent
- replay buffer 数据结构
- tool calling 接口
- 状态机与资源池

### 10.3 创新设计层
在 MedAgentSim 基础上，重点考虑以下创新：

- 不确定性感知分诊
- 多医生争议仲裁
- 失败模式仪表盘
- 资源感知型医院环境
- 安全监督与风险拦截
