# 清华模拟医院（Agent Hospital / MedAgent-Zero）文献梳理与后续问答总结

## 一、文档目的

本文档用于系统总结“清华模拟医院”相关论文与代码仓库的核心内容，并整理围绕该工作的后续问答，作为后续模拟医院多智能体系统（Medical MAS）项目设计的参考依据。

本文档重点包括：

- 对 **Agent Hospital / MedAgent-Zero** 论文的系统梳理
- 对其方法、工作流、memory 与 evolution 机制的分析
- 对其代码仓库实现完整性与 baseline 可行性的检查
- 对相关前端技术（Tiled / Phaser / Streamlit / Gradio）的理解
- 对该工作相较前一篇 **MedAgentSim** 的互补价值分析
- 对本项目的直接指导意义总结

---

## 二、论文基本定位

该论文的核心目标不是仅做静态医疗问答，而是模拟一个**医生在医院中长期看诊、积累经验并持续进化**的过程。论文认为，真实医生的成长并不只依赖书本知识，而是依赖在医院里通过病例实践获得经验，因此需要一个“医院模拟世界”来支持医生 agent 的长期进化。

相较于很多只关注 MedQA/USMLE 等考试题 benchmark 的医疗 LLM 工作，这篇论文更强调：

- 医生如何在模拟医院环境中工作
- 如何通过持续接触 patient agent 获得经验
- 如何将虚拟世界中的成长迁移到真实 benchmark 上

---

## 三、Motivation / Observation

### 3.1 核心动机

论文指出，现有医疗大模型研究主要集中在“医学知识获取”，例如通过考试题、问答任务来测试模型是否掌握医学知识。但真实医生的能力还包含第二阶段，即：

- 在医院中处理真实病例
- 在长期实践中积累经验
- 从失败与反思中成长

因此，仅靠静态医学 QA，并不能模拟医生真实成长过程。

### 3.2 关键观察

论文的关键观察包括：

1. **LLM 本身并不等价于医生。**  
   真正像医生的是“在环境中行动、规划、反思、调用工具”的 LLM-powered agent。

2. **现有多 agent 医疗系统多关注协作推理，而较少关注长期成长。**  
   很多系统能做 doctor-patient 对话或多医生讨论，但尚未真正模拟“在医院中通过持续看病不断变强”的机制。

3. **需要一个模拟医院世界来支持进化。**  
   如果能构造一个医院 simulacrum，让医生反复诊治 patient agents，并记录成功与失败经验，就能实现某种形式的“虚拟世界训练”。

## 四、Contribution / Innovation

论文的主要创新可概括为以下几点。

### 4.1 构建了医院全闭环模拟世界

论文不只模拟问诊，而是尽可能覆盖完整医院流程，包括：

- 发病
- 分诊
- 挂号
- 问诊
- 检查
- 诊断
- 取药
- 康复
- 复诊

这使得系统从“诊疗交互器”扩展为“医院级闭环模拟环境”。

### 4.2 提出了 SEAL 范式

论文提出 **SEAL（Simulacrum-based Evolutionary Agent Learning）**，由两部分组成：

- **Simulacrum Construction**：构建医院模拟世界并自动生成 patient 数据
- **Agent Evolution**：让 doctor agent 在模拟世界中通过实践持续进化

### 4.3 提出了 MedAgent-Zero

MedAgent-Zero 是医生进化机制，核心思想为：

- 成功案例进入 **medical case base**
- 失败案例经过反思、验证、精炼后进入 **experience base**

这说明系统不仅存储病例，还会从错误中抽取可复用经验。

### 4.4 证明虚拟世界训练具有迁移性

论文展示了 doctor agent 在虚拟医院中不断看诊后，其在：

- examination selection
- diagnosis
- treatment plan recommendation

三类任务上的表现持续提升，并且这种提升还能迁移到真实 MedQA benchmark。

## 五、Method / Workflow

### 5.1 总体 workflow

论文整体流程可概括为：

1. 利用知识库自动生成 patient agents  
2. patient 在医院中经历完整流程：分诊、挂号、问诊、检查、诊断、取药、康复、复诊  
3. doctor 根据病例结果判断自己是否成功治疗  
4. 成功则写入 case base，失败则反思、生成规则、验证后写入 experience base  
5. doctor 持续读书、持续接触更多 patient，从而不断进化

这一流程的重点在于“长期成长”，而不是单轮问答。

---

### 5.2 LLM

论文把 LLM 当作不同角色 agent 的“大脑”，包括：

- patient agent
- nurse agent
- doctor agent

其中，doctor 所依赖的 base model 是**冻结的**，并不会做参数微调。系统的进化主要依赖外部 memory 与经验库，而不是直接训练模型参数。

这对 API 驱动项目具有直接借鉴意义，因为课程项目通常无法真正更新大模型参数，更适合采用：

**冻结大模型 + 外挂案例库 / 经验库** 的路线。

---

### 5.3 Tools

论文未显式以函数签名形式给出 tools，但从系统机制上看，已有明确的工具链：

- 分诊
- 挂号
- 检查
- 药房取药
- 医学知识库调用
- 病例库写入
- 经验库写入

因此从工程角度，可以将其抽象成：

- `triage()`
- `register()`
- `request_exam()`
- `dispense_medicine()`
- `retrieve_knowledge()`
- `write_case_base()`
- `write_experience_base()`

---

### 5.4 Memory

论文的 memory 设计非常关键，主要包括两类。

#### 5.4.1 Medical Case Base
保存成功病例，用于后续检索和 RAG。

#### 5.4.2 Experience Base
保存失败后经反思、验证、精炼得到的经验规则。

与普通 replay 不同，这里并非仅保存失败记录，而是把失败上升为“可复用规则”。

这一点对于项目设计非常有启发性，因为它说明长期 memory 可以进一步区分为：

- 成功案例库
- 失败修正规则库

---

### 5.5 Planning

论文的 planning 更偏“事件驱动闭环”，而非单轮 ReAct。

医院流程由多个事件组成：

- Disease Onset
- Triage
- Registration
- Consultation
- Medical Examination
- Diagnosis
- Medicine Dispensary
- Convalescence

此外还有医生的 **Reading Books** 事件。

因此，该工作实际上更接近：

**状态 / 事件 / 下一个状态**

而不是简单的 “Thought → Action → Observation”。

---

### 5.6 MCP / Skills 的映射理解

论文没有显式使用 MCP 术语，也未将 skills 模块化，但可自然映射为：

- **Skills**：分诊、检查选择、诊断、治疗建议、复诊跟踪
- **Memory**：case base + experience base
- **Planning**：医院事件闭环
- **Tools**：知识库检索、检查结果读取、案例写入、经验写入
- **Environment**：医院地图、功能区、事件触发

因此，它非常适合作为课程项目“高层系统框架”的参考。

## 六、是否有前端交互

有，而且前端可视化程度较强。

论文明确说明，其医院 sandbox simulation 使用：

- **Tiled**：绘制医院地图与功能区
- **Phaser**：构建浏览器中的 2D 仿真界面

医院被划分为多个功能区，例如：

- 分诊台
- 挂号处
- 候诊区
- 诊室
- 检查室
- 药房
- 复诊区

这说明该工作并非纯 CLI 系统，而具有比较强的“医院场景感”。

## 七、后端实现理解

论文中的后端核心逻辑可概括为四层。

### 7.1 Sandbox Environment
基于 Tiled + Phaser 构建医院地图与环境。

### 7.2 Patient Generation Pipeline
给定疾病后，自动生成：

- 基本信息
- 病史
- 症状
- 检查结果

### 7.3 Doctor Evolution Engine
负责：

- 检索 case base / experience base
- 写入成功案例
- 对失败案例进行反思、验证、精炼
- 更新经验库

### 7.4 Knowledge Base Coupling
通过医学知识库与公开医学案例对 patient 生成过程进行约束，并通过质量控制 agent 提升病例合理性。

因此，该工作的后端重点不在传统业务系统工程，而在于：

**模拟生成 + 记忆增强 + 经验进化**

论文最终呈现了三类结果。

### 8.1 系统层结果
展示一个完整的医院 simulacrum，patient 从发病到分诊、挂号、问诊、检查、诊断、取药、康复和复诊的全过程。

### 8.2 虚拟世界层结果

- medical examination selection
- diagnosis
- treatment plan recommendation

## 十一、与前一篇 MedAgentSim 的互补关系

这部分是该论文对项目最有价值的地方。

### 11.1 MedAgentSim 的强项

前一篇 MedAgentSim 的强项在于：

- doctor / patient / measurement 三方交互清晰
- doctor 主动补问与申请检查
- 多轮交互更像真实门诊
- experience replay 机制清晰
- 有公开代码，复现较友好

### 11.2 MedAgentSim 的短板

但它更像一个“单次临床交互模拟器”，仍存在不足：

- 医院流程范围较小
- 长期进化视角较弱
- 分诊、药房、康复、复诊等环节不够完整
- replay 更偏记录与反思，没有发展成成熟的规则库

### 11.3 Agent Hospital 对其补强

Agent Hospital 正好在以下方面形成补充：

#### （1）从单次诊疗扩展到长期成长
强调 doctor 在长期实践中不断进化。

#### （2）从 conversation-level 扩展到 hospital-level
把 triage、registration、medicine dispensary、convalescence、follow-up 纳入闭环。

#### （3）从 replay 扩展到规则化经验积累
不仅记录失败，还从失败中提炼规则并验证后入库。

#### （4）从一次性评测扩展到成长曲线与迁移分析
展示 agent 随患者数量增加而持续提升，并迁移到真实 MedQA 上。

### 11.4 最合理的结合方式

因此，两篇工作最优的结合方式不是二选一，而是：

- 用 **MedAgentSim** 提供 doctor-patient-measurement 的底层交互骨架
- 用 **Agent Hospital** 提供医院闭环、长期经验积累、规则化反思和成长曲线这几层创新

## 十二、代码仓库检查：https://github.com/wisdom-pan/Agent_Hospital

### 12.1 仓库整体判断

这个仓库是一个可运行的多 agent 医疗问诊原型，适合作为轻量级 baseline。  
但它不是论文中那种完整医院级模拟器，也不是成熟的前后端工程系统。

更准确地说，它属于：

**多 agent 医疗对话系统 + 结果展示器**

而不是：

**完整医院沙盘模拟平台**

---

### 12.2 做到了什么

#### （1）有明确的多 agent 结构
仓库 `src/agents` 下包含：

- `base_agent.py`
- `doctor.py`
- `patient.py`
- `reporter.py`
- `host.py`

说明系统至少支持：

- doctor
- patient
- reporter / examiner
- host / 协作控制

#### （2）有多种模型后端封装
在 `engine` 层中可以看到对多种模型引擎的封装，便于接入不同 API。

#### （3）有单医生与多医生协作两类场景
`src/hospital` 下有：

- `consultation.py`
- `collaborative_consultation.py`

表明系统既支持单 doctor 问诊，也支持多医生讨论与 critique 机制。

#### （4）有评测与输出模块
仓库还带有：

- `src/evaluate`
- `src/scripts`
- JSONL 输出
- 运行脚本与评估脚本

因此它不是空壳，而是一个可运行研究原型。

---

### 12.3 它是怎么工作的

其核心流程可以概括为：

1. doctor 发起问候或提问  
2. patient 回复  
3. 如果对话中需要检查，则 reporter / examiner 根据 medical records 返回检查结果  
4. doctor 继续整合信息  
5. 最终输出：

- 症状
- 辅助检查
- 诊断结果
- 诊断依据
- 治疗方案

多医生模式下，则额外加入 host 组织并行讨论与 critique。

---

### 12.4 前端完整性

仓库有前端，但前端能力较弱。

#### （1）使用 Gradio
`src/app.py` 使用的是 Gradio。

#### （2）本质更像离线结果回放器
其界面主要从已有输出 JSONL 中读取病例，逐步展示 doctor / patient / reporter 对话和 chief physician 结果。

因此它更像：

**离线演示播放器**

而不是：

**真正在线交互的医院前端系统**

#### （3）没有明显前后端分离结构
仓库中没有：

- `frontend/`
- `backend/`
- API server
- 数据库 schema
- 前端现代工程结构

所以其前端不属于完整业务系统前端。

---

### 12.5 后端完整性

#### 优点
仓库已具备：

- agent 抽象层
- engine 抽象层
- scenario 抽象层
- 运行脚本
- 基本评测模块

#### 局限
但仍缺少：

- 医院资源管理层
- 排队与床位调度
- 事件队列 / 状态机
- 权限与审计

因此它更像：

**多 agent 问诊实验框架**

而不是：

**医院级仿真平台**

---

### 12.6 作为 baseline 的可靠性

#### 正面因素
- 公开可访问
- 有 README、requirements、scripts、evaluate、outputs
- 有一定 stars 与 forks
- 具备多 agent、多模型、多场景运行痕迹

#### 保留意见
- 提交历史较少，工程成熟度一般
- README 较简略
- 前端偏演示性质
- 缺少医院系统关键模块

#### 结论
它是：

**研究 demo 层面可靠**

但不是：

**工程产品层面可靠**

---

### 12.7 作为 baseline 的适用范围

#### 适合作为：
- 多 agent 问诊 baseline
- 多医生协作诊断 baseline
- reporter / examiner 机制 baseline
- 多模型兼容调用 baseline

#### 不适合作为：
- 医院环境模拟 baseline
- 完整前后端产品 baseline
- 资源感知调度 baseline
- 状态机 / 事件驱动 hospital OS baseline

因此，更合理的用法是：

**把这个仓库当作模块级 baseline，而不是整系统 baseline。**

## 十三、后续问答总结

### 13.1 什么是 schema

schema 指的是：

**对数据结构或系统对象格式的明确规定**

即定义：

- 应该有哪些字段
- 字段类型是什么
- 字段之间如何组织

在模拟医院 MAS 中，schema 特别重要，因为它决定了：

- 多个 agent 如何通信
- 前后端如何对接
- 检查结果如何返回
- replay buffer 如何设计
- 模型输出如何校验

---

### 13.2 Tiled 和 Phaser 是什么

#### Tiled
Tiled 不是 Python 库，而是一个 **2D 地图编辑器**，主要用于绘制医院地图、功能区和对象图层。

#### Phaser
Phaser 是一个 **JavaScript / TypeScript 的 2D 前端游戏框架**，用于把 Tiled 画好的地图真正做成交互式网页界面。

#### 是否能用 Python 直接实现
不能把它们简单理解为 Python 界面库。更合理的分工是：

- Python：后端 agent、workflow、状态机、LLM 调用
- Tiled + Phaser：前端可视化地图与动态场景

因此它们需要单独学一点前端开发，而不是纯 Python 开发。

---

### 13.3 Streamlit / Gradio 与 Phaser 的区别

#### Streamlit / Gradio
更适合：

- AI demo
- 数据与日志展示
- agent workflow 面板
- 文本输入输出
- 图表、表格、状态面板

优点：
- 全 Python
- 开发快
- 和 LLM agent 后端衔接方便

缺点：
- 空间感弱
- 不擅长做地图式医院仿真
- 容易更像“管理后台”而不是“医院沙盘”

#### Phaser
更适合：

- 医院地图
- 功能区可视化
- 患者 / 医生流转
- 排队状态
- 动画与事件驱动展示

优点：
- 演示效果强
- 非常适合“模拟医院”主题

缺点：
- 学习成本更高
- 需要 JS / TS
- 需要和 Python 后端额外对接
- 开发周期更长

#### 对项目的建议
最稳妥的路线是：

- 第一阶段用 **Streamlit / Gradio** 完成主系统界面
- 第二阶段若时间允许，再加 **Phaser 小地图模块** 做增强展示

## 十四、对本项目的直接指导意义

综合论文、仓库和后续问答，可以得出以下结论。

### 14.1 可直接借鉴的内容

- 医院闭环流程设计思路
- case base + experience base 双重长期记忆
- 从失败中抽取规则并验证的经验进化机制
- 多 agent 问诊协作与 critique 模式
- 分阶段构建前端：先 AI demo，再考虑地图仿真

### 14.2 不应直接照搬的内容

- 仅把系统做成论文级概念框架而缺少可运行代码
- 只做问诊对话而忽略医院环境层
- 只做 benchmark accuracy，而忽略流程、资源和安全

### 14.3 对你们项目最合理的路线
